# 贡献指南

感谢您对五子棋游戏项目的关注和贡献！以下是参与本项目的指南。

## 代码贡献流程

1. **克隆仓库**：从GitHub克隆项目到本地
2. **创建分支**：为您的功能或修复创建一个新的分支
3. **开发**：实现您的功能或修复
4. **测试**：确保您的修改不会破坏现有功能
5. **更新文档**：每次修改代码后，确保更新README.md文件以反映您的更改
6. **提交代码**：提交您的更改，确保提交信息清晰明了
7. **推送到远程**：将您的分支推送到GitHub
8. **创建Pull Request**：提交PR并描述您的更改

## 重要规则

### 必须遵守的规则

1. **README更新规则**：每次push代码前，必须更新README.md文件，确保文档与代码保持同步
2. **代码风格**：遵循PEP 8代码风格指南
3. **测试**：确保您的修改通过所有测试
4. **提交信息**：使用清晰、描述性的中文提交信息
5. **分支管理**：使用有意义的分支名称
6. **推送前确认**：必须先向用户确认推送内容，得到同意后再执行推送操作

### README更新要求

- 当添加新功能时，在功能特性部分添加说明
- 当修改项目结构时，更新项目结构部分
- 当更改游戏操作流程时，更新游戏操作部分
- 当添加新的技术实现时，更新技术实现部分
- 确保所有文档内容准确反映当前代码状态

## 功能开发规则

### 新增功能
1. 确保新增功能与现有代码风格一致
2. 为新增功能添加适当的注释
3. 测试新增功能是否正常工作
4. 更新相关文档（如果需要）

### 代码修改
1. 理解现有代码结构和逻辑
2. 保持修改的最小化和针对性
3. 测试修改是否影响其他功能
4. 确保修改后的代码可以正常运行

## 测试规则

### 功能测试
- 每次修改后应测试相关功能是否正常
- 测试应覆盖常见使用场景
- 确保游戏可以正常启动、运行和退出

### 边界测试
- 测试边界情况，如网络连接失败、异常输入等
- 确保程序在异常情况下能够优雅处理

## 项目结构规则

### 文件组织
- 源代码文件存放在 `src/` 目录下
- 主入口文件为根目录的 `main.py`
- 保持项目结构清晰，避免不必要的文件和目录

### 命名规范
- 文件名使用小写字母和下划线，例如：`game_mode_selector.py`
- 类名使用驼峰命名法，例如：`GameModeSelector`
- 函数和变量名使用小写字母和下划线，例如：`start_single_player`

## 代码风格指南

- 使用4个空格进行缩进
- 每行不超过79个字符
- 使用清晰的变量和函数命名
- 添加适当的注释
- 遵循Python的命名约定

## 提交信息规范

提交信息应遵循以下格式：

```
<类型>: <描述>

<详细说明>（可选）
```

类型包括：
- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码风格修改
- refactor: 代码重构
- test: 测试相关
- chore: 构建或依赖更新

### 中文提交信息示例

```
feat: 添加布局切换功能

实现了垂直布局、水平布局和网格布局三种界面风格，用户可以在游戏开始前选择喜欢的布局。
```

```
fix: 修复网络对战连接问题

解决了客户端连接主机时的超时问题，提高了网络对战的稳定性。
```

```
docs: 更新README.md文档

添加了布局切换功能的说明，更新了项目结构和游戏操作流程。
```

## 操作流程

### 日常开发流程
1. 阅读本规则文件
2. 检查当前代码状态
3. 进行必要的修改
4. 测试修改内容
5. 提交修改（如果需要）
6. 推送前向用户确认
7. 执行推送操作（如果用户同意）

### 紧急修复流程
1. 阅读本规则文件
2. 定位问题所在
3. 进行最小化的修复
4. 测试修复效果
5. 提交修复（如果需要）
6. 推送前向用户确认
7. 执行推送操作（如果用户同意）
