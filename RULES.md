# 五子棋项目操作规则

## 1. 代码管理规则

### 1.1 代码修改流程
1. **修改前**：确认当前分支状态，确保在正确的分支上进行修改
2. **修改中**：遵循项目现有的代码风格和命名规范
3. **修改后**：测试修改是否正常工作
4. **提交前**：检查修改内容，确保没有引入无关变更
5. **推送前**：
   - 更新README.md文件（如果需要）
   - 必须先向用户确认推送内容，得到同意后再执行推送操作

### 1.2 Git操作规范
- 提交信息应清晰明确，使用中文描述变更内容
- 提交信息格式：`类型: 描述`，例如：`Feature: 添加新功能`、`Fix: 修复bug`
- 每次推送前必须先确认推送内容

## 2. 功能开发规则

### 2.1 新增功能
1. 确保新增功能与现有代码风格一致
2. 为新增功能添加适当的注释
3. 测试新增功能是否正常工作
4. 更新相关文档（如果需要）

### 2.2 代码修改
1. 理解现有代码结构和逻辑
2. 保持修改的最小化和针对性
3. 测试修改是否影响其他功能
4. 确保修改后的代码可以正常运行

## 3. 测试规则

### 3.1 功能测试
- 每次修改后应测试相关功能是否正常
- 测试应覆盖常见使用场景
- 确保游戏可以正常启动、运行和退出

### 3.2 边界测试
- 测试边界情况，如网络连接失败、异常输入等
- 确保程序在异常情况下能够优雅处理

## 4. 项目结构规则

### 4.1 文件组织
- 源代码文件存放在 `src/` 目录下
- 主入口文件为根目录的 `main.py`
- 保持项目结构清晰，避免不必要的文件和目录

### 4.2 命名规范
- 文件名使用小写字母和下划线，例如：`game_mode_selector.py`
- 类名使用驼峰命名法，例如：`GameModeSelector`
- 函数和变量名使用小写字母和下划线，例如：`start_single_player`

## 5. 操作流程

### 5.1 日常开发流程
1. 阅读本规则文件
2. 检查当前代码状态
3. 进行必要的修改
4. 测试修改内容
5. 提交修改（如果需要）
6. 推送前向用户确认
7. 执行推送操作（如果用户同意）

### 5.2 紧急修复流程
1. 阅读本规则文件
2. 定位问题所在
3. 进行最小化的修复
4. 测试修复效果
5. 提交修复（如果需要）
6. 推送前向用户确认
7. 执行推送操作（如果用户同意）

## 6. 注意事项

- 保持代码风格一致
- 确保代码可维护性
- 及时更新文档
- 推送前必须确认
- 测试是确保质量的关键
